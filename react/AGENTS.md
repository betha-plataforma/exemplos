# Repository Guidelines

## Project Structure & Module Organization

This is a Vite React TypeScript app. Application code lives in `src/`.

- `src/main.tsx` bootstraps React and registers Betha web components.
- `src/App.tsx` defines routing, shell layout, and Betha header/menu integration.
- `src/pages/` contains route-level UI modules. Use PascalCase filenames for pages, for example `Dashboard.tsx`.
- `src/pages/shared.tsx` contains small reusable page helpers.
- `src/data/examples.ts` contains static demo data and exported TypeScript types.
- `src/custom-elements.d.ts` and `src/estrutura-componentes-loader.d.ts` provide local typings for Betha custom elements.
- `public/` contains static assets served as-is. `src/assets/` contains bundled assets.
- `dist/` is generated by builds and should not be edited manually.

## Build, Test, and Development Commands

- `npm run dev`: start the Vite development server.
- `npm run build`: run `tsc --noEmit` and then create a production build with Vite.
- `npm run lint`: run ESLint across the repository.
- `npm run preview`: serve the production build locally after `npm run build`.

There is currently no `npm test` script or test framework configured.

## Coding Style & Naming Conventions

Use TypeScript and TSX for source files. Keep strict type checking clean; do not replace useful types with broad `any` unless there is no practical alternative. Follow the existing style: 2-space indentation, single quotes, no semicolons, and named exports for pages/components except the root `App` default export.

Use PascalCase for React components and page files, camelCase for local variables/functions, and uppercase snake case for exported static constants such as `MENU_OPTIONS`.

## Testing Guidelines

For now, validation is linting and production build type checking:

```sh
npm run lint
npm run build
```

When adding a test framework, prefer colocated tests named `*.test.ts` or `*.test.tsx`, and cover route behavior, filters, modals, and Betha custom-element integration points.

## Commit & Pull Request Guidelines

Recent history uses conventional-style prefixes such as `feat:` and `chore:`. Keep commits short and imperative, for example `feat: add list filters` or `chore: update dependencies`.

Pull requests should include a concise summary, verification commands run, linked issue when applicable, and screenshots for visible UI changes.

## Security & Configuration Tips

Do not commit real access tokens or credentials. Demo authorization values in `src/data/examples.ts` should remain sample-only. Keep Betha custom-element typings synchronized with actual component usage when changing refs, events, or assigned properties.
